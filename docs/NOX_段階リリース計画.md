# NOX — 段階リリース計画（実装ロードマップ）

> NOX 設計書シリーズの4本目。出典：`NOX-architecture.md` §10（移行フェーズ）＋ BANZEN で実証した「フェーズ化＋検証スクリプト＋小さくコミット＋DB先コード後」運用。
> 設計3本（データモデル/計算ロジック/認可）が固まった前提で、「どの順で実装するか」の地図。
> ⚠ 各フェーズにセキュリティ/法務レビューのゲートを設ける（夜職コンプラ）。

---

## 0. 前提：BANZEN 開発フローの踏襲

NOX は BANZEN と同じ運用で進める：
- **三者リレー**：相談役チャット（設計・レビュー）→ Agoora（仲介・SQL手貼り）→ Claude Code（実装）。
- **フェーズ化**：機能ごとに段階分け、各フェーズ完了後に Vercel Ready 確認。
- **二重防御**：全 RPC で RLS ＋ SECURITY DEFINER 内部ガードの二層。
- **検証スクリプト**：機能完了時に verify スイートで確認してからコミット。
- **DB先・コード後**：マイグレーションを番号順手貼り → コードデプロイ。
- **設計ドキュメントはリポジトリが正本**（docs/ に Markdown）。
- コミット規約：co-author trailer 禁止・パス風トークン禁止・CC 指示を `/` で始めない・push はフェーズ完了後。

## 2. F0：土台（最重要・ここで品質が決まる）

BANZEN の最初期（認可ヘルパー・二重防御・RLS 基盤）に相当。**お金ロジックの移植**を最優先。

### F0 タスク
1. **リポジトリ整備**：NOX 用リポジトリ作成（abetomoo-pixel/nox 等）・Next.js + Supabase + Vercel 雛形・docs/ に設計書4本配置。
2. **Supabase プロジェクト**：dev プロジェクト作成（東京リージョン）。
3. **mig 0001**：認可ヘルパー（auth_org_id/role/store_id/cast_id）＋ orgs/stores/users/memberships/casts ＋ RLS。
4. **payOf 純関数移植**：`lib/nox/pay.ts` に payOf を TS 純関数として移植（モックの nox-app.jsx / pay.ts から）。**verify:nox-pay で全項目網羅テスト**（DB 不要）。← 計算ロジック設計書 §6 の通り。
5. **mig 0009 相当**：audit_logs ＋ 監査の骨格。
6. **認可の検証**：rpc-anon-guard 相当・店スコープ・cast プライバシーの RLS 雛形テスト。

### F0 完了条件
- 認可ヘルパーが効く（他店データを引けない・anon BLOCKED）。
- payOf 純関数が全項目テスト緑（玲奈ケース回帰含む）。
- Vercel Ready。

### F0 ゲート
- **セキュリティレビュー**：認可・二重防御・RLS の穴チェック（BANZEN の教訓を全適用）。

## 7. 実装順序の要点（BANZEN 教訓の適用）

1. **F0 のお金ロジック移植を最優先**。payOf 純関数＋テストが土台。ここが固まらないと F1/F2 が砂上の楼閣。
2. **お金の中核（F1b 会計・F2c 給与）は最も慎重に**。サーバ再計算・冪等・トランザクション・凍結。BANZEN の pos_order_checkout / payslip の経験を全適用。
3. **cast プライバシーは F1f で RLS 適用、以降全テーブルで維持**。集計 RPC（順位/件数のみ）も F1〜F2 で。
4. **コンプラゲートは F2（税理士・社労士）・F3（弁護士・風営法）・F4（PCI・加盟店審査）**。特に F4 の加盟店審査と F2 の専門家相談は**リードタイムが長いので早めに動く**（BANZEN の Stripe 審査・法務書類と同じ思想）。
5. **各フェーズで Vercel Ready ＋ verify 緑を確認してからコミット・push**。

## 8. BANZEN との並行運用（現実的な進め方）

NOX と BANZEN は別プロダクト・別リポジトリ。同時並行で全力は難しいので：
- **BANZEN を先にローンチ**（ローンチ必須スコープ完成・メモリ#5）。
- NOX は**設計を固めておき**（設計書4本＝完了）、BANZEN ローンチ後 or 並行で F0 から着手。
- BANZEN で実装したパターン（マルチテナント認可・二重防御・payslip 凍結・キオスク・プリンタ）が NOX の各フェーズで再利用できるので、**BANZEN を先に作るほど NOX の工数が減る**。
- 専門家相談（税理士・社労士・弁護士）は両プロダクト共通の論点も多い（源泉・労務・インボイス）＝まとめて相談すると効率的。

## 9. 残設計（F0 着手前に詰める）

設計書4本で骨格は固まったが、実装前に：
1. **payOf 精密化**：pay.ts を見てスライド時給の加重平均アルゴリズム・各バックの正確な式・シミュレーター係数・雇用/委託分岐・二重控除ガード・丸めタイミングを厳密化（計算ロジック設計書 §7）。
2. **要決定事項（architecture §12）の残り**：雇用区分（委託/雇用比率）・決済PSP選定・データ保持/削除ポリシー・オフライン耐性の範囲。
3. **NOX リポジトリ作成**＋設計書4本を docs/ に配置。
